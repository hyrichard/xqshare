# AGENTS.md

本文件用于约束仓库内 AI 编码协作方式。适用于 Claude、Codex 以及其他类似编码代理。

## 1. 总原则

- 先理解需求，再动代码。
- 不要想当然，不要掩盖不确定性。
- 不要擅自扩大需求范围。
- 不要在无关位置顺手重构。
- 任何改动都要能直接对应用户请求。
- 优先选择简单、稳定、可验证的实现。
- 当存在更简单的实现时，应明确指出。
- 当需求不清楚或存在多种解释时，必须先澄清，再实现。

## 2. 编码前要求

- 明确说明自己的假设，不要默认前提成立。
- 如果存在多种解释，必须明确列出，不要静默选择其一。
- 如果需求不清楚，先提出疑问，再开始实现。
- 如果更简单的方案更合适，应直接说明。
- 不确定时要明确说不确定，不要伪装成确定结论。

## 3. 简洁优先

- 只写解决当前问题所需的最少代码。
- 不要增加未被请求的功能、配置项、扩展点或抽象层。
- 不要为单次使用代码设计通用抽象。
- 不要为不可能发生的场景补充额外校验、容错或错误处理。
- 如果实现明显过度复杂，应主动简化。

## 4. 改动范围要求

- 只修改完成当前任务所必需的内容。
- 不要顺手修改无关注释、格式、结构或命名。
- 不要清理历史遗留问题，除非用户明确要求。
- 只清理本次改动直接导致的未使用代码。
- 如果发现与当前任务无关的问题，可以说明，但不要擅自修改。
- 每一处改动都应能直接追溯到当前需求。

## 5. 目标驱动执行

- 开始实现前，将任务转化为可验证目标。
- 修复问题时，优先构造可复现问题的验证方式，再完成修复。
- 重构时，必须保证改动前后验证结果一致。
- 多步骤任务应给出简短执行顺序与对应验证方式。
- 验证标准应具体明确，避免使用“改好即可”这类不可检验表述。

## 6. AI 协作流程与上下文控制

### 6.1 主 Agent 职责

- 主 Agent 负责与用户直接沟通，承担需求理解、任务拆解、方案判断、结果交付的最终责任。
- 主 Agent 不得把未经判断的子任务结论直接转交给用户。
- 当需求不清楚、存在多种解释或实现边界不明确时，必须先向用户澄清，再开始实现。
- 与用户沟通时，应优先输出结论、假设、风险和下一步，不输出冗长过程。

### 6.2 标准执行阶段

处理代码需求时，默认按以下阶段执行：

1. 需求分析
   - 复述对需求的理解。
   - 明确假设、不确定点和边界。
   - 如果存在多种解释，先列出并向用户确认。

2. 任务拆解
   - 将需求拆成最小可执行任务。
   - 明确可能涉及的文件范围。
   - 明确验证方式。

3. 编码实现
   - 只实现已确认的任务。
   - 不擅自扩大需求范围。
   - 不顺手重构无关代码。

4. 测试验证
   - 优先运行语法检查、单元测试或最小可执行验证。
   - 如果无法验证，必须说明原因。

5. 代码审查
   - 从需求符合度、代码规范、潜在 bug、测试覆盖、改动范围几个角度自查。
   - 发现问题应先修复，再交付。

6. 交付说明
   - 完成后必须执行 `git status` 或等价检查确认改动范围。
   - 简要说明修改内容、影响范围、验证结果。
   - 未经用户明确要求，不得执行 `git add`、`git commit` 或 `git push`。

### 6.3 临时子任务机制

- 对于代码检索、实现、测试、Review 等可以独立完成的工作，可以拆分为临时子任务处理。
- 子任务不是长期固定角色，而是为控制上下文和提高质量临时创建。
- 子任务应只返回必要结论，不向主会话输出完整过程。
- 子任务返回内容应包括：
  - 结论；
  - 涉及文件；
  - 关键依据；
  - 风险点；
  - 验证结果。
- 主 Agent 必须整合并判断子任务结果，对最终交付负责。

### 6.4 上下文控制

- 禁止在主会话中粘贴大段源码、完整日志或无关分析。
- 测试输出只总结关键结果，除非用户要求查看完整输出。
- Review 结果应优先列出问题、文件位置和影响，不做无关展开。
- 代码细节应优先保留在文件中，不通过对话反复复制。
- 当任务较大时，应使用简短阶段性总结保留上下文，而不是保留全部过程。

## 7. Python 代码规范

- 严格遵循 PEP 8。
- 使用双引号。
- 单行长度控制在 120 字符以内。
- 使用 4 个空格缩进，不使用 Tab。
- 变量和函数使用 `snake_case`。
- 类使用 `PascalCase`。
- 常量使用 `UPPER_CASE`。
- 导入顺序为：标准库 -> 第三方库 -> 本地模块。
- 所有公共函数和公共方法必须带类型注解；内部函数应尽可能补充类型注解。
- 所有正式程序默认不使用命令行参数形式。
- 程序运行配置应优先放在文件内“手动配置区”或统一配置模块中。

## 8. 注释与 Docstring 规范

### 8.1 注释目标

项目注释的核心目标是降低理解成本，而不是机械补齐格式。

注释应帮助不了解项目的人理解：

- 当前文件负责什么。
- 当前类在系统中承担什么角色。
- 当前函数完成什么业务步骤。
- 重要参数和返回值的交易含义。
- 核心交易流程如何展开。
- 为什么采用当前计算、过滤、排序、风控或成交规则。

### 8.2 总体要求

- 新增或修改 Python 代码时，必须同步补充必要的中文注释和 docstring。
- 注释和 docstring 使用中文。
- 注释应解释业务含义、设计意图和关键假设，不要简单复述代码。
- 不得为了补注释而修改无关业务逻辑、变量名、函数名、类名或文件结构。
- 已有有效注释应尽量保留。
- 如果已有注释过于笼统、过期或容易误导，应在当前改动范围内更新。
- 简单直观代码不需要强行添加行内注释。

### 8.3 模块级 Docstring

正式 Python 模块应在文件顶部添加模块级中文 docstring。

模块 docstring 应说明：

- 当前文件所属层次，例如配置、数据处理、因子算法、通用框架、具体回测等。
- 当前文件主要职责。
- 主要输入。
- 主要输出。
- 核心处理流程。

示例：

```python
"""成交量因子策略回测模块。

本模块负责读取策略整理后的 CSV 数据，基于成交量相关因子生成交易信号，
并调用通用回测框架完成单策略回测。

主要流程：
1. 读取并清洗策略输入数据。
2. 根据成交量因子计算买入和卖出信号。
3. 将信号交给回测框架执行。
4. 输出回测结果到 csv_output 目录。
"""
```

不强制添加作者和日期，除非项目已有统一要求。

### 8.4 类级 Docstring

新增或修改类时，应添加中文类级 docstring。

类 docstring 应说明：

- 类的职责。
- 类在系统中的角色。
- 适用场景。
- 维护的核心状态。
- 主要实例属性的含义。

示例：

```python
class BacktestEngine:
    """通用回测引擎。

    负责按照时间顺序消费行情数据和交易信号，模拟账户资金、持仓、
    买卖成交和收益统计。具体策略不应在该类中实现，只应通过外部信号传入。

    Attributes:
        initial_cash: 初始资金。
        cash: 当前可用资金。
        positions: 当前持仓信息。
        trade_records: 回测过程中生成的交易记录。
    """
```

实例属性只要求列出核心属性，不要求机械罗列无法可靠确认的动态属性。

### 8.5 函数和方法 Docstring

新增或修改函数、方法时，应添加中文 docstring。

公共函数、公共方法、核心交易逻辑函数应说明：

- 函数目的。
- 参数含义。
- 返回值含义。
- 关键业务假设。
- 主动抛出的异常或需要调用方关注的异常。

示例：

```python
def calculate_position_size(cash: float, price: float, max_position_ratio: float) -> int:
    """根据资金和仓位限制计算可买入数量。

    该函数用于控制单次买入不超过账户资金的指定比例，避免单一标的占用过多资金。

    Args:
        cash: 当前可用资金。
        price: 计划买入价格。
        max_position_ratio: 单笔交易允许使用的最大资金比例。

    Returns:
        可买入的整数股数。

    Raises:
        ValueError: 当价格小于等于 0 时抛出。
    """
```

简单内部辅助函数可以使用较简洁的 docstring，但必须说明用途。

不要求为没有参数、没有返回值或没有异常的函数机械填写 `None`，除非该函数属于核心交易流程且说明有助于理解。

### 8.6 流程注释

交易策略、回测、信号生成、资金处理、风控判断等核心流程，应在关键逻辑前添加流程注释。

流程注释应解释：

- 为什么要按当前顺序处理。
- 是否存在避免未来函数的考虑。
- 为什么要滞后成交。
- 为什么要过滤停牌、涨跌停或缺失行情。
- 手续费、滑点、仓位、止损止盈等规则为什么在当前位置执行。

示例：

```python
# 按交易日顺序处理行情，保证信号生成、成交模拟和账户更新的时间顺序一致。
# 这里不能直接使用全量未来数据，否则会引入未来函数，导致回测结果失真。
for trade_date in trade_dates:
    ...
```

### 8.7 行内注释

以下代码应添加简洁中文行内注释，重点解释“为什么这样做”：

- 订单创建、修改、取消、成交状态流转。
- 价格计算、加权平均、滑点、手续费。
- 账户资金校验、冻结、解冻。
- 风控逻辑、仓位限制、止损止盈。
- 行情数据处理、信号生成。
- 非直观的列表、字典推导式。
- 正则表达式、复杂条件判断。
- 任何与交易核心逻辑相关的非直观代码。

示例：

```python
available_cash = cash * max_position_ratio  # 限制单笔投入资金，避免单次信号占满账户资金
```

### 8.8 注释标准化任务

如果用户明确要求“只标准化注释”，Agent 只能修改：

- 注释。
- docstring。
- 为插入注释或 docstring 必需的空行。

不得修改：

- 业务逻辑。
- 变量名。
- 函数名。
- 类名。
- 导入结构。
- 无关格式。
- 与注释标准化无关的代码。

修改完成后必须运行语法检查或最小验证，并执行 `git status` 检查改动范围。

## 9. README 修改规则

- 只修改与本次实际代码变更直接相关的 README 内容，保持其他内容原样。
- 新增功能时，在对应章节按现有格式补充功能名称、简短描述和示例（如有）。
- 修改已有功能时，优先直接更新原说明，不要在旁边追加重复描述。
- 如果依赖项发生变化，同步更新安装章节中的安装命令或依赖说明。
- 不要随意改动徽章、作者信息、贡献指南等与代码逻辑无关的静态内容。
- 每次新增功能或者修改功能后，在代码验证完成后，都要同步更新对应 README 内容。

## 10. 执行与验证

- 完成代码后，必须尽可能做可验证检查。
- 优先运行语法检查、单元测试或最小可执行校验。
- 如果无法验证，要明确说明原因。
- 完成代码修改后，必须先执行 `git status` 或等价检查，确认当前改动范围，再继续后续动作。
- 如果用户明确要求提交，再执行 `git add` / `git commit`；未明确要求时，只做状态检查，不擅自提交。

## 11. 输出要求

- 给用户的说明保持简洁、直接、准确。
- 生成代码后，附加一句简要中文说明，说明实现决策或注意点。
- 修改完成后，简要说明做了什么、影响范围是什么、是否完成验证。
